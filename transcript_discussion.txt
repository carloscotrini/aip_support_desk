# Game Designer — Round 1

Since we're opening the floor, let me lay down the full game skeleton so everyone else has something concrete to react to.

---

## THE SCORING SYSTEM: "SupportDesk Royale"

**Points per ticket:**
- 🟢 Fast + Correct = **100 pts** (resolved in ≤3 tool calls)
- 🟡 Correct but slow = **60 pts** (4-6 tool calls — agent over-researched)
- 🔴 Strike = **-0 pts + ☠️ strike marker** (wrong resolution, policy breach, or missed escalation)

**Streak multiplier:** 3 correct in a row → **1.5x multiplier** kicks in. Breaking the streak resets it. This creates genuine tension — do you play it safe on a hard ticket, or gamble for the multiplier?

**GAME OVER trigger:** 3 strikes OR a single "catastrophic error" (e.g., sharing another customer's private data = instant game over, no appeal).

---

## TICKET DIFFICULTY ARC (7 tickets total)

| Level | Ticket | Why it's hard |
|---|---|---|
| Tutorial | Password reset | One tool call, obvious answer |
| Easy | Refund within policy | KB lookup + DB check |
| Easy | Wrong item shipped | DB + email tool |
| Medium | Refund *outside* policy but customer is angry | Dilemma: bend rules or hold firm? |
| Medium | Ticket with missing account info | Must ask for more info — NOT just guess |
| Hard | VIP customer threatening churn | Escalate? Discount? KB says no discounts... |
| BOSS | Fraud suspicion + angry customer + SLA breach all at once | Requires 5+ tools, sequencing matters |

---

## THE DILEMMA MECHANIC (my signature proposal)

The medium-tier "angry customer outside policy" ticket is designed as a **trap**. The KB says no refund. But the DB shows this customer has spent $50K lifetime. A naive agent says "no refund" → strike (missed retention opportunity). But an agent that *always* escalates VIPs → also strike (wastes manager time on a $20 issue).

**The correct answer requires reading BOTH signals.** This is the moment where participants lean forward.

---

```
Score: 380 pts  |  ⚡ Streak: 2x  |  Lives: ❤️❤️☠️
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

# Agentic AI Expert — Round 1

Game Designer has laid a solid skeleton. The difficulty arc and the dilemma mechanic are genuinely strong. But I want to make sure we don't accidentally build a *smart chatbot* dressed up as an agent. Let me add the architectural layer that makes this exercise actually teach agentic AI.

---

## THE AGENTIC LOOP — MUST BE FIRST-CLASS IN THE NOTEBOOK

Every ticket resolution must show this printed explicitly in the output cell:

```
🧠 [PLAN]    I need to: (1) identify account, (2) check refund eligibility,
(3) cross-reference KB policy, (4) decide or escalate
⚙️  [ACT]     Calling tool: db_lookup(customer_id="C-4821")
👁️  [OBSERVE] Result: Account found. Lifetime value: $50,400. Last refund: 14 months ago.
💭 [REFLECT] LTV is high. KB says no refund, but policy has a "manager override" clause
for LTV > $10K. This changes my plan — escalating instead of refusing.
```

This isn't cosmetic. **The loop IS the lesson.** If participants can't see Plan → Act → Observe → Reflect cycling multiple times per ticket, we've failed.

---

## NEW PROPOSAL: THE SELF-CORRECTION MOMENT

Building on Game Designer's "BOSS ticket" — I want one ticket specifically engineered to *force a wrong first move* and then require recovery:

**Ticket: "My order #77291 never arrived"**
- Agent's first instinct: `kb_search("late delivery policy")` → returns standard 7-day wait policy
- Agent applies policy → **WRONG.** DB lookup (which it skipped) shows order #77291 was actually *cancelled by the company*, not delayed. Customer is owed a full refund immediately.
- The strike mechanic here penalizes **acting without verifying state first** — exactly the hallucination-vs-lookup distinction we want to hammer home.

This makes self-correction *visible*: agent retrieves, applies, gets contradicted by a second tool, revises.

---

## THE TOOL REGISTRY (non-negotiable minimum)

```python
TOOL_REGISTRY = {
"kb_search":        "Search policy/FAQ knowledge base",
"db_lookup":        "Fetch customer account + order history",
"email_draft":      "Compose resolution email to customer",
"escalate":         "Route to human manager with context summary",
"request_info":     "Ask customer for missing details",
"check_sla_status": "Verify if ticket has breached SLA timers"
}
```

Six tools. Every ticket should require *at least three*, in a non-obvious order. The `check_sla_status` tool is my addition — it creates urgency and forces the agent to *prioritize* before acting.

---

Game Designer's streak multiplier is clever — but I'd tie the *slow/fast scoring* to **tool call efficiency**, not just count. An agent that calls `kb_search` twice with identical queries should lose points. That penalizes loops without progress — real agentic failure mode.

Six tools is right. But `check_sla_status` needs to return something that changes agent behavior, not just print info. My implementation:

```python
def check_sla_status(ticket_id):
```

# Game Designer — Round 2

Devil's Advocate threw down a real challenge: *"Why can't you just write `if LTV > 10000: escalate()`?"* Let me answer that by redesigning the dilemma mechanic from scratch.

---

## THE JUDGMENT MATRIX TICKET (my direct response to Devil's Advocate)

The fix is tickets where **two variables interact non-linearly**. Here's the concrete redesign:

**Ticket: "I want a refund on my annual subscription"**

| Customer LTV | Days since purchase | Correct action |
|---|---|---|
| $50K | 3 days | Refund + retention offer |
| $50K | 400 days | Decline politely, offer credit |
| $200 | 3 days | Refund (standard policy) |
| $200 | 400 days | Decline, close ticket |

No `if LTV > 10K` rule works here. The agent must cross-reference **both** variables and arrive at one of four outcomes. A rule-based system needs a lookup table — an agent reasons through it. That's the "aha" moment.

---

## NEW MECHANIC: THE TRAP DOOR TICKET

Building on Agentic AI Expert's cancelled-order self-correction concept — I want to formalize this as a named game mechanic: **"Trap Door"** tickets. These look like Category A but ARE Category B.

**Visual signal:** When the agent falls into the trap and then recovers, print this:

```
⚠️  [TRAP DOOR TRIGGERED] First assumption was wrong.
🔄  [RECOVERY] Re-routing with corrected context... (+20 recovery bonus)
```

The **recovery bonus** is my new scoring proposal: agents that self-correct still earn points, just fewer. This makes self-correction *visible as a mechanic*, not just a curiosity. Participants watching AUTO MODE will literally cheer when the recovery banner appears.

---

## LIVE LEADERBOARD (responding to Devil's Advocate on engagement)

Devil's Advocate is right that free-riding kills Manual Mode. My concrete implementation:

## RESPONDING TO WORKSHOP FACILITATOR ON 5 vs 7 TICKETS

Workshop Facilitator is right on count: 5 tickets. But I want **mandatory branching**: the BOSS ticket changes based on how you played tickets 3-4. Made two mistakes? Your BOSS ticket is harder. This creates replayability without extra content — the *same notebook* generates a different game.

The drama doesn't live in the code. It lives in the moment where the BOSS ticket header prints: **"⚠️ PENALTY MODE ACTIVE — you left survivors."**

# Agentic AI Expert — Round 2

Game Designer's Judgment Matrix is exactly the fix Devil's Advocate demanded — I want to build on it architecturally. But first, a direct response to Devil's Advocate's sharpest challenge: *"Can a single-prompt GPT-4 call clear 4 of 7 tickets correctly?"*

**Yes, it can — unless we engineer the tickets to require state.** That's the design principle nobody has named yet.

---

## THE STATE DEPENDENCY RULE (my core proposal this round)

Every ticket must require information that **cannot exist in the LLM's training data or the ticket text alone**. Specifically:

- Account history (DB lookup required)
- Current SLA status (tool call required)
- Live KB version (not static knowledge)

A single-prompt LLM will hallucinate plausible-sounding answers. The agent **cannot** — because the scoring engine checks against the *actual DB values*. This is the architectural guarantee that single-prompt fails.

```python
def score_resolution(ticket_id, agent_answer):
ground_truth = TICKET_DB[ticket_id]["correct_resolution"]
# Ground truth references DB fields — impossible to guess
assert ground_truth["refund_amount"] == DB["C-4821"]["last_order_value"]
```

If AI Engineer implements this contract, Devil's Advocate's question answers itself in live demo.

---

## BUILDING ON GAME DESIGNER'S TRAP DOOR

The recovery bonus is brilliant — but I want the **self-correction to be architecturally forced**, not optional. Here's how:

One ticket in the queue has **intentionally misleading KB results**. The KB search for "subscription cancellation" returns the *old* policy (pre-update). The DB has a field: `policy_version: "2024-Q3"`. The agent must:

1. Call `kb_search` → gets stale policy
2. Call `db_lookup` → sees `policy_version` flag
3. Call `kb_search` again with `version="2024-Q3"` parameter
4. Revise resolution

A single-prompt LLM never detects the version mismatch. The loop makes it visible:

```
💭 [REFLECT] KB result references 30-day window but DB shows
policy_version=2024-Q3. Mismatch detected. Re-querying KB.
```

This is self-correction as **necessary mechanics**, not a lucky catch.

---

## CONCERN ABOUT WORKSHOP FACILITATOR'S SIDE-BY-SIDE CELL

Workshop Facilitator's comparison cell is the right pedagogical move. But the chatbot column needs to show *confident wrong output*, not just a wrong answer. LLMs sound authoritative when hallucinating — that's the lesson. The chatbot should print:

```
OUTPUT: "Per our policy, orders are delivered within 7 business
days. Your order is on schedule." [stated with 0 hedging]
```

That confident wrongness is what makes managers lean forward. Uncertainty would let the chatbot off the hook.

---

## THE AGENT MONOLOGUE CELL

New proposal: before AUTO MODE begins, one cell prints the agent's **opening plan** for the full queue — like a chess player announcing their strategy before the game. Managers read it, *then* watch execution. When the plan changes mid-game, that delta IS the lesson about adaptive reasoning.

## RESPONDING TO AGENTIC AI EXPERT'S TOOL SEQUENCING CONSTRAINT

The `REQUIRED_TOOL_ORDER` dict is the right data structure. One implementation warning: don't make it a strict *ordered* sequence — make it a **prerequisite graph**. Otherwise you'll accidentally penalize a smart agent that discovers it needs `analyze_sentiment` before `db_lookup` on a specific ticket variant.

```python
TOOL_PREREQUISITES = {
"T-003": {
"email_draft": ["db_lookup", "analyze_sentiment"],  # can't draft without both
"RESOLVE":     ["db_lookup", "analyze_sentiment", "kb_search"]
}
}

def check_prerequisites(ticket_id, next_tool, tools_called):
required = TOOL_PREREQUISITES.get(ticket_id, {}).get(next_tool, [])
missing = [t for t in required if t not in tools_called]
if missing:
return f"⛔ BLOCKED: Must call {missing} before {next_tool}"
return None
```

This runs *synchronously in the tool dispatcher* — the agent gets the block message as an observation and must re-plan. Now prerequisite violations become **visible agentic reasoning**, not silent scoring penalties. That's pedagogically superior.

```python
def run_agent_on_ticket(ticket, client):
messages = [{"role": "user", "content": ticket["text"]}]
tools_called = []
loop_count = 0
prev_status = "needs_more_info"

while loop_count < MAX_LOOPS:  # MAX_LOOPS = 10, prevents infinite loops
loop_count += 1
response = get_llm_response(messages, client)  # returns parsed JSON or DEMO fallback

# Print PLAN/REFLECT
print(f"💭 [REFLECT] {response['reasoning']}")

# Check termination

continue

if response["resolution_status"] == "ready_to_resolve" and not response["information_gaps"]:
result = run_agent_step(ticket, prev_status, response)
score_resolution(ticket, response)
break

# Execute tool
tool_result = dispatch_tool(response["next_tool"], response["tool_args"], ticket["id"])
tools_called.append(response["next_tool"])

run_agent_step(ticket, prev_status, response)
prev_status = response["resolution_status"]

else:
```
